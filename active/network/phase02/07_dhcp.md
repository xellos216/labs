# Objective

Understand how DHCP supplies the basic network configuration that lets a
Linux host participate in a network.

This session focused on DHCP as a configuration mechanism rather than only
an IP-address assignment mechanism.

---

# Learning Summary

DHCP provides a host with the information it needs to join a network. It can
provide an IP address, subnet or prefix, default gateway, DNS servers, and
lease information.

The routing table showed routes marked with `proto dhcp`, meaning the routing
entries were installed from configuration learned through DHCP. The address
output showed a dynamic IPv4 address with remaining valid and preferred
lifetimes, connecting the observed address to the idea of a DHCP lease.

The DNS investigation also showed that receiving DNS information from DHCP and
using DNS at the system resolver layer are related but separate steps. The
physical interface had DNS servers from NetworkManager, while the final
`/etc/resolv.conf` file was generated by an overlay/VPN resolver. This showed
that Linux network configuration may pass through several layers before an
application performs name resolution.

# Practical Observations

## Observation 1 - Dynamic Address

Prediction:

```text
The interface address was likely assigned automatically by DHCP.
```

Observation:

```text
inet 203.0.113.10/24 brd 203.0.113.255 scope global dynamic noprefixroute eth0
    valid_lft 6236sec preferred_lft 6236sec
```

Explanation:

```text
203.0.113.10/24
        ↓
IPv4 address and prefix
        ↓
dynamic
        ↓
Address came from dynamic network configuration
        ↓
valid_lft / preferred_lft
        ↓
Lease-style lifetime information is visible
```

The important point is that the address is not treated as permanently owned by
the host. It has a validity lifetime.

## Observation 2 - DHCP-Installed Routes

Prediction:

```text
The gateway route should show evidence that it came from DHCP configuration.
```

Observation:

```text
default via 203.0.113.1 dev eth0 proto dhcp src 203.0.113.10 metric 100
```

Explanation:

```text
DHCP configuration
        ↓
Network manager / DHCP client
        ↓
Kernel routing table
        ↓
default route installed with proto dhcp
```

This connected DHCP to the earlier routing and default-gateway sessions. DHCP
did not only assign an address; it also supplied the gateway information used
for off-LAN traffic.

## Observation 3 - DNS from Interface Configuration

Prediction:

```text
The interface should have DNS server information associated with it.
```

Observation:

```text
IP4.ADDRESS[1]: 203.0.113.10/24
IP4.GATEWAY:    203.0.113.1
IP4.DNS[1]:     198.51.100.53
IP4.DNS[2]:     198.51.100.54
```

Explanation:

```text
DHCP / NetworkManager result
        ↓
IP address
Gateway
DNS servers
```

This confirmed that DHCP can provide a configuration bundle, not just an IP
address.

## Observation 4 - Resolver Layer Can Differ

Prediction:

```text
The resolver file should show which DNS settings the system is currently using.
```

Observation:

```text
# resolv.conf file generated by an overlay/VPN resolver
nameserver 100.100.100.100
search example.ts.net example.local
```

Explanation:

The interface had DNS information from NetworkManager, but the final
`/etc/resolv.conf` file was generated by another resolver-management layer.

```text
DHCP
        ↓
NetworkManager receives interface DNS
        ↓
System resolver configuration is generated
        ↓
Overlay/VPN software may replace the final resolver file
        ↓
Applications use the final resolver configuration
```

This showed that DHCP-provided DNS and actually applied system DNS are not
always the same visible layer.

---

## Observation 5 - Missing Configuration Cases

Prediction:

```text
Different missing network settings should break different kinds of communication.
```

Explanation:

```text
IP + subnet available
        ↓
Same-LAN communication can work

IP + subnet available, but no default gateway
        ↓
Same-LAN communication can work
Off-LAN communication fails

IP + subnet + gateway available, but no DNS
        ↓
External IP communication can work
Domain-name communication may fail
```

This clarified that DHCP provides several independent pieces of configuration.
Each piece supports a different layer of network participation.

# Commands / Code

```bash
ip -4 addr show eth0
```

Shows the IPv4 address, prefix, dynamic status, and address lifetime for an
interface.

---

```bash
ip route
```

Shows routes installed in the kernel routing table. Routes marked with
`proto dhcp` were installed from DHCP-learned configuration.

---

```bash
cat /etc/resolv.conf
```

Shows the resolver configuration currently visible to applications that use the
standard resolver path.

---

```bash
nmcli dev show eth0 | grep -E 'IP4.ADDRESS|IP4.GATEWAY|IP4.DNS'
```

Shows the IPv4 address, gateway, and DNS values associated with a
NetworkManager-managed interface.

# Common Misconceptions

- Incorrect: DHCP only gives an IP address.
- Correct: DHCP can provide an IP address, subnet or prefix, default gateway,
  DNS servers, and lease information.

---

- Incorrect: `resolvectl` failure means DNS does not exist.
- Correct: It may only mean `systemd-resolved` is not managing DNS on that
  system.

---

- Incorrect: DNS information from DHCP and `/etc/resolv.conf` must always match.
- Correct: Another resolver-management layer, such as an overlay or VPN client,
  may replace the final resolver file.

---

- Incorrect: A host needs a default gateway for all communication.
- Correct: Same-LAN communication can work with only a correct IP address and
  subnet or prefix.

---

- Incorrect: If DNS is missing, all network communication fails.
- Correct: IP-based communication can still work, while domain-name resolution
  may fail.

---

# Key Takeaways

- DHCP is a network configuration distribution mechanism.
- DHCP commonly provides IP address, subnet or prefix, default gateway, DNS
  servers, and lease time.
- `proto dhcp` in the routing table indicates DHCP-learned route configuration.
- A dynamic address with `valid_lft` and `preferred_lft` connects to DHCP lease
  behavior.
- DNS has multiple layers: interface DNS, resolver management, and final
  application-visible resolver configuration.
- Same-LAN communication, off-LAN communication, and domain-name communication
  depend on different pieces of configuration.
